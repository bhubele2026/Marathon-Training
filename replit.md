# Marathon Command Center

## Overview

Personal training tracker reframed as a general workout planner. Originally a 1-year half-marathon campaign (281.6 lbs → 210 lbs by race day **2027-05-02** Sun, plan starts **2026-05-04** Mon, 52 weeks across Foundation Build / Aerobic Build / Tempo-Threshold / Race-Specific / Taper & Race). The planner now also supports Tonal-first / non-running programs: 4 lift-priority templates (`tonal_strength_upper`, `tonal_strength_lower`, `push_pull_legs`, `tonal_conditioning`), an 8-week Tonal upper-body default for new configs, and a "Training for a marathon?" toggle on the Config card that gates the legacy 16-week MARATHON_TAIL auto-pin. Race-only UI surfaces (Race Campaign header, Race Day countdown, "13.1 mi") fall back to a generic "Workout Plan" framing when the active plan has no Marathon-Specific phase. Each week runs Mon → Sun.

UI is an orange-on-dark "mission control" theme. No emojis anywhere.

The HR Zone targeting mode supports four zone models — `five_zone_max` (default), `friel_7_zone`, `coggan_5_zone`, `polarized_3_zone` — picked from a dropdown on Settings. Per-model zone tables, labels, and bucket→zone mapping live in `artifacts/command-center/src/lib/run-target.ts` (`HR_ZONE_MODEL_DEFS`).

## Weekly schedule (lift-priority, hybrid)

Generated by `scripts/src/generate-plan.ts`. Rules:

- **Mon** — full REST (optional walk/mobility).
- **Tue** — Heavy upper-body Tonal + 20–25 min Peloton Bike (short cardio).
- **Wed** — Easy Tread run + light Tonal core/accessory.
- **Thu** — Heavy lower-body Tonal + 20–25 min Peloton Row (short cardio).
- **Fri** — Quality Tread run (tempo/threshold/race-pace once past Foundation phase). On big run days: NO lift. In Foundation phase (W1–6) it's just an easy run + light accessory.
- **Sat** — Heavy full-body Tonal + 20–25 min Peloton Bike or Row (alternates).
- **Sun** — LONG RUN, no lift. Race week (W52): race day.

Bike/row sessions are always 20–30 min max. Big run days (Fri quality, Sun long, race) carry no lift. Most days mix lifting with short cardio for ≥1 hour total. Lifting volume is intentionally heavy.

## Pages

- `/` Dashboard — mission status, weekly snapshot, body mass, today's mission, recent activity, equipment usage.
- `/today` Today's mission with three quick actions (Crushed It / Log Actual / Skipped). Before the campaign begins (today < first non-rest plan day), shows a "Pre-Launch — Campaign starts in N days" countdown card with a preview of the first scheduled session in lieu of the regular plan brief. Powered by `daysUntilStart` + `firstSession` on `/api/plan/today`.
- `/plan` 52-week plan grouped by phase. Weekly summary cards render a per-program completion ratio breakdown (Task #162, "Tonal Lift 1/2 · 5K Improver 1/2") under the combined adherence bar whenever the week has 2+ overlapping programs; data comes from `PlanWeek.programs` on `/api/plan/weeks`. The same `programs[]` shape is also returned by `/api/plan/weeks/:week` and surfaced under the Sessions tile on `/plan/:week`. The dashboard snapshot per-program block adds an Adherence cell (campaign-to-date %) sourced from `DashboardSummary.programs[].adherencePct` so a runner can spot which program drags overall adherence down. Two reset controls: **Reset Entire Plan** (top right, restores plan-day prescriptions to the seeded snapshot — preserves logged workouts and measurements, undoable for ~30s) and **Full Reset** (Danger Zone card at the bottom, requires typing `WIPE EVERYTHING` to confirm — TRUNCATEs every workout/measurement/checklist/customization and reseeds the campaign from day one, no undo).
- `/plan/:week` Week detail with 7 day cards. Each non-rest card has three quick actions: **Crushed It** (one-click logs a workout matching the planned values + hype toast), **Log Actual** (opens the form pre-filled), **Skipped** (confirm dialog → logs `sessionType="Skipped"` + roast toast). Sass copy lives in `src/lib/sass.ts`; mutation logic in `src/hooks/use-mission-actions.tsx`. Skipped workouts are excluded from adherence/completion counts and weekly mileage in `dashboard.ts` / `plan.ts`.
- `/log` Workouts list with equipment filter + log dialog. The Log Workout dialog has a **Cardio or Strength** modality picker plus an **Equipment / Machine** picker grouped by modality (Cardio: Peloton Tread/Bike/Row/Outdoor · Strength: Tonal · Other: Lifestyle / None). Modality is auto-inferred from equipment if blank, never overwriting an explicit user choice. Persisted on `workouts.modality` (nullable text) and propagated by Crushed It.
- `/measurements` Body weight trend + check-in dialog.
- `/equipment` Per-equipment session counts and recent activity.

## Data

- Plan + days + body baselines seeded from the most-recently-applied planner config via `pnpm --filter @workspace/scripts run seed`. On a fresh install (no applied config) the seed script inserts a default stacked config named **"Tonal Upper 8wk"** (entries-mode, single `tonal_strength_upper` template, 8 weeks anchored on the next Monday) and seeds the plan rows from it; legacy `.local/data/plan.json` is still regenerated for out-of-band tooling but no longer drives the seed.
- The `/plan` and `/dashboard` headers and the sidebar nav label all read from `overview.activeConfigName` / `summary.activeConfigName` (Task #244) — whatever the runner named their planner config — falling back to "Workout Plan" when no config has ever been applied. Race-only subtitles ("Days/Weeks to Race Day") still gate on `raceKind`.
- Tables (lib/db): `plan_weeks`, `plan_days`, `workouts`, `body_measurements`, `race_week_checklist` (default item state + user-added custom items, distinguished by `is_custom` + nullable `label`).
- Seed reseeds plan_weeks / plan_days on every run, wipes `workouts` (campaign reset), and preserves `body_measurements` and `reset_undo_snapshots`.
- `plan_days` carries a three-bucket minute breakdown (`strength_min`, `cardio_min`, `run_min`) plus matching `seed_*` mirrors. `cardio_min` is non-running cross-train minutes only (bike, row, spin); treadmill / outdoor running minutes live in `run_min`. The API exposes `totalMin` (server-computed sum) so the `/today` and `/plan/:week` workout cards render a stable TOTAL · LIFT · CARDIO · RUN tile via `<PlannedBreakdown>`. The one-shot backfill script `pnpm --filter @workspace/scripts run backfill-plan-day-minutes` populates the new columns from the canonical generator output and runs automatically on every merge via `scripts/post-merge.sh`.
- `race_results` table (one row keyed by `race_date`) holds the runner's logged race-day result — finish time, overall + total placement, 1–5 felt rating, free-form notes, and split `recorded_at` / `updated_at` timestamps. The `/api/race-week` endpoint embeds the row as `raceResult` only after `racePassed` flips true so the post-race banner can swap the recovery copy for a "Log your race" form (and back to a saved-result summary). Wiped by Full Reset.
- `plan_days` also carries an `equipment_list` chip rail (text array) plus matching `seed_equipment_list` mirror. The list is the ordered set of every machine the runner uses that day so `/plan/:week` day cards and the `/today` Mission Brief / Pre-Launch First Scheduled Session render one chip per machine (e.g. "TONAL · PELOTON BIKE" for a Tue strength + cardio day) instead of just the headline scalar. **Contract:** the scalar `equipment` column always equals `equipment_list[0]` (the *primary* machine for that day) so any back-compat code path that still reads the scalar — dashboard equipment usage, suggestions pairKey, `/equipment` page — agrees with the chip rail's lead chip. The generator emits `["Tonal", "Peloton Tread"]` for Wed and the foundation Fri (W1-6), with scalar `equipment = "Tonal"`. PATCH on `/api/plan/:date` collapses the chip rail to `[equipment]` whenever the runner edits the scalar through the form, preserving the contract. The backfill script `pnpm --filter @workspace/scripts run backfill-plan-day-equipment` populates the column for any pre-task row whose chip rail is still NULL by building `[scalar equipment, ...secondary machines mentioned in the description, in order of appearance]`, deduped — so legacy rows keep their existing scalar at index 0 and the contract holds. The script is idempotent and runs automatically on every merge.

## Stack

- pnpm workspaces, Node 24, TypeScript 5.9
- Express 5, Drizzle ORM, Postgres
- Zod (`zod/v4`), `drizzle-zod`
- OpenAPI codegen via Orval (`pnpm --filter @workspace/api-spec run codegen`)
- React 18 + Vite, wouter, @tanstack/react-query, shadcn/ui, recharts, date-fns

## Key commands

- `pnpm run typecheck` — typecheck everything. Also wired up as a validation step so it runs before tasks are marked complete.
- `pnpm run test` — run every workspace package's `test` script (currently the api-server vitest suite). Also wired up as a validation step so it runs before tasks are marked complete.
- `pnpm --filter @workspace/api-server run test` — run just the api-server vitest suite
- `pnpm --filter @workspace/api-spec run codegen` — regenerate API hooks/zod from `lib/api-spec/openapi.yaml`
- `pnpm --filter @workspace/db run push` — push schema (dev)
- `pnpm --filter @workspace/scripts run seed` — reseed plan + baseline measurements

## Deployment notes

- Personal-use app: no auth/authz layer; do not expose the published URL publicly.
- CORS is permissive by default — restrict via env if needed.
- "Today" is computed in UTC; tasks near midnight may shift by a day.

See the `pnpm-workspace` skill for monorepo conventions.
